Crear un proyecto
	Ir a la carpeta que va a contener a la Carpeta [NOMBRE_PROYECTO]
		Es decir, la línea de abajo crea la carpeta [NOMBRE_PROYECTO] y pone todos los archivos adentro
	npx create-next-app [NOMBRE_PROYECTO]

	No es obligación trabajar con TS 

	Poner `src/` directory = "Sí" cuando los proyectos sean relativamente grandes y puedan ir creciendo

	App Router: Yes

	TurboPack: Yes

	Customize import alias: No
	
Ejecutar un proyecto
	npm run dev

Layouts distintos para diferentes páginas
	Puedo poner un layout.tsx dentro de la carpeta donde quiero modificar el diseño. Ese layout es un HOC, donde el children es la página que quiero renderizar.
		En ese HOC tengo que implementar el children donde lo quiera renderizar

	Si quiero tener un solo layout para múltiples carpetas puedo tener esta estructura de archivos:

	/
	|_app
	     |_(general)
			|_about
			|_contact
			|_pricing
			|_layout.tsx // <--- Este layout impacta sobre las tres carpetas. Se le pone (general) para que no agregue "general" a la URL
	     |_layout.tsx	// Layout y
	     |_page.tsx		// Home page

Generación estática y revalidación (5/69)
	Se pueden generar de antemano, estáticamente, páginas dinámicas que reciben un ID, para servirlas estáticamente.
		Esa generación se produce en Build Time (npm run build)

	Para eso, se ejecuta una función que se llama generateStaticParams, la cual devuelve un array estático con los parámetros que espera la página dinámica para que genere dinámicamente en tiempo de construcción todas esas páginas estáticas.

	En el ejemplo 5/62 la página dinámica espera el parámetro id, entonces mi función generateStaticParams va a ser así:
		export async function generateStaticParams() {
		  return [
		    { id: '1' },
		    { id: '2' },
		    { id: '3' },
		    { id: '4' },
		    { id: '5' },
		    { id: '6' }
		  ];

	Al tener esto, y ejecutar npm run build, de antemano Next genera las páginas estáticas para los casos donde se reciban parámetros del 1 al 6, entonces no tiene que volver a generarlas cuando el usuario las visite. En caso que el usuario ingrese id: 7, se genera esa página también.

	En este ejemplo tenemos:
		try {
		    const pokemon = await fetch(`https://pokeapi.co/api/v2/pokemon/${id}`, {
		      //cache: 'force-cache'
		      next: {
		        revalidate: 60 * 60 * 24 * 30 * 6
		      }
		    }).then(resp => resp.json());

	Esto hace que Next revalide cada X segundos (60 * 60 * 24 * 30 * 6 = 6 meses). Funciona porque Next, al momento de servir la página rastrea si existen fetchs que se deban revalidar. Si no hay que revalidar, sirve la versión actual, si expiró validación la regenera y sirve la nueva versión.

	Si se quiere revalidar una página independientemente del fetch se puede poner esta instrucción fuera de la exportación del componente:
		export const revalidate = 60 * 60 * 30 * 6;
